# SmartNote 语雀迁移版需求文档

版本：V1.0  
文档日期：2026-04-11  
文档类型：产品需求文档（PRD）  
适用范围：SmartNote Web 端、后端 API、数据库模型

## 1. 文档目标

本文档用于将“从语雀迁移用户视角出发的优化建议”收敛为可执行、可排期、可验收的正式需求。

本文档只定义一个可交付版本的范围，不覆盖所有长期设想。目标是先让 SmartNote 从“智能笔记工具”升级为“可承接语雀迁移的知识库产品”。

## 2. 背景与现状

### 2.1 当前已具备能力

当前项目已具备以下核心能力：

- Markdown 编辑与实时协同编辑
- 笔记本、笔记、标签、回收站、历史版本
- 单篇笔记分享、段落级评论、评论处理页
- AI 摘要、AI 标签建议、AI 问答
- 知识图谱、导出 Markdown/PDF/Word
- 文件夹分组能力的后端与部分前端实现

### 2.2 当前核心短板

虽然已有较完整的“笔记工具能力”，但距离“语雀替代品”仍存在结构性缺口：

- 缺少导入能力，用户无法低成本迁移存量内容
- 缺少完整的多级知识库目录树
- 缺少知识库级公开发布，只支持单篇分享
- 缺少成员与权限体系，协作主要依赖分享链接

以上四项会直接影响用户是否愿意将主知识库迁入 SmartNote，因此必须优先解决。

## 3. 版本目标

### 3.1 版本目标

V1.0 的目标不是“补更多功能”，而是完成以下产品跃迁：

- 用户可以将外部文档批量迁入 SmartNote
- 用户可以在 SmartNote 内组织成可维护的知识库目录树
- 用户可以把整个知识库公开发布，而不是只分享单篇
- 用户可以邀请成员协作，并基于角色进行访问控制

### 3.2 成功标准

本版本上线后，应满足以下验收结果：

- 用户可在 10 分钟内完成一个中小型知识库的迁入与结构恢复
- 用户可在一个知识库内创建、调整、浏览至少 3 级目录
- 用户可公开发布一个知识库，并通过目录页浏览多篇内容
- 用户可为一个知识库添加至少 3 类成员角色，并验证权限差异

## 4. 范围定义

## 4.1 本版本纳入范围

- FR-01 导入中心
- FR-02 多级目录树
- FR-03 知识库公开发布
- FR-04 成员与权限

## 4.2 明确不纳入本版本

以下能力不属于本版本范围，避免项目失焦：

- AI 深度编辑能力（选中改写、扩写、自动结构化）
- 搜索引擎重构（全文索引、容错、权重优化）
- 双链、反向链接、文中提及关系图
- 通知中心、消息收件箱、@提及
- 移动端专项重构
- 表格数据库、任务管理、看板视图

以上内容可进入 V1.1 或后续版本评估，不在本 PRD 中立项。

## 5. 用户角色定义

### 5.1 角色列表

- 知识库所有者：创建知识库、管理成员、发布知识库、拥有最高权限
- 知识库管理员：管理目录、文档、成员，但不能转移所有权
- 编辑者：可新建、编辑、移动、删除自己有权限的内容
- 评论者：可阅读并发表评论，不可编辑文档
- 访客：仅可阅读公开内容

### 5.2 迁移用户画像

目标用户为“已在语雀中维护文档/知识库，并希望迁移到 SmartNote 的个人用户或小团队”。

此类用户的核心诉求不是单篇编辑，而是：

- 迁移成本低
- 目录结构可控
- 协作权限清晰
- 可作为正式知识库对外发布

## 6. 功能需求

## 6.1 FR-01 导入中心

### 6.1.1 目标

让用户可以将外部文档批量导入 SmartNote，作为迁移入口能力。

### 6.1.2 功能范围

V1.0 仅支持以下导入方式：

- 单个 Markdown 文件导入
- Markdown ZIP 包批量导入
- 单个 Word 文档导入

V1.0 不支持直接调用语雀开放平台 API，不支持 Notion、Confluence 等第三方平台导入。

### 6.1.3 入口要求

- 在知识库首页提供“导入”入口
- 在知识库详情页提供“导入到当前知识库”入口
- 导入过程必须显示进度状态

### 6.1.4 功能要求

- 用户可选择导入目标知识库
- 导入 Markdown 文件时，系统自动创建一篇笔记
- 导入 ZIP 包时，系统按 ZIP 目录结构自动创建文件夹与笔记
- 导入 Word 文件时，系统将正文转换为 Markdown 后创建笔记
- 导入图片资源时，系统将图片上传至本地文件存储，并替换文档中的引用路径
- 导入完成后，系统输出结果摘要，包括成功数量、失败数量、失败原因

### 6.1.5 业务规则

- 单次导入文件总大小上限：100MB
- 单个 ZIP 包最大文件数：1000
- 文件名为空时，系统使用原文件名或“未命名文档-时间戳”作为标题
- 导入失败不得影响已成功导入的文档
- 重名文件夹允许存在，但同一目录下重名笔记需自动追加序号

### 6.1.6 数据要求

需要补充或确认以下模型能力：

- 导入任务记录表
- 导入任务状态字段：`PENDING / PROCESSING / SUCCESS / PARTIAL_SUCCESS / FAILED`
- 导入日志明细表或结构化失败列表

### 6.1.7 验收标准

- 导入单个 Markdown 文件后，用户可在目标知识库内看到对应笔记
- 导入 ZIP 包后，目录结构与 ZIP 内层级一致
- 导入包含图片的 Markdown 时，图片可正常显示
- 导入失败时，页面能显示具体失败文件与原因
- 100 篇以内的批量导入可在合理时间内完成，不出现整批回滚

## 6.2 FR-02 多级目录树

### 6.2.1 目标

将当前“根笔记 + 单层文件夹分组”升级为真正的多级知识库目录树。

### 6.2.2 功能范围

V1.0 支持最多 5 级目录嵌套。

### 6.2.3 目录对象定义

目录树由两类节点组成：

- 文件夹节点
- 笔记节点

知识库根节点不显示为单独节点，但作为树结构的默认挂载点。

### 6.2.4 功能要求

- 用户可创建顶级文件夹
- 用户可在任意文件夹下创建子文件夹
- 用户可在根目录或任意文件夹下创建笔记
- 用户可通过拖拽移动文件夹位置
- 用户可通过拖拽移动笔记到目标文件夹
- 用户可展开/折叠任意文件夹
- 用户可重命名文件夹
- 用户可删除文件夹
- 用户可在页面中始终看到当前笔记的目录路径（面包屑）

### 6.2.5 排序规则

- 同层级下文件夹优先于笔记显示
- 同层级节点支持手动排序
- 若未手动排序，则按创建时间升序显示

### 6.2.6 删除规则

- 删除文件夹时，不删除其下文档
- 删除文件夹时，其下笔记与子文件夹整体提升到上一级目录
- 删除根级文件夹时，其下内容移动到根目录

### 6.2.7 业务约束

- 单知识库最大目录深度：5
- 单知识库最大节点数：5000
- 禁止将文件夹拖入自身或其后代目录中

### 6.2.8 数据要求

需要补充或确认以下字段：

- 文件夹表：`parent_folder_id`
- 文件夹表：`sort_order`
- 笔记表：`folder_id`
- 笔记表：`sort_order`（当前若无，需新增）

### 6.2.9 验收标准

- 用户可创建至少 3 级嵌套目录并正常浏览
- 拖拽移动笔记或文件夹后，刷新页面结构不丢失
- 删除文件夹后，其内部笔记不会丢失
- 打开任意笔记时，页面可显示正确面包屑路径

## 6.3 FR-03 知识库公开发布

### 6.3.1 目标

将当前“单篇分享”升级为“知识库级公开发布”，让 SmartNote 可承接正式文档站点场景。

### 6.3.2 功能范围

V1.0 支持“单知识库公开发布”，不支持自定义域名，不支持主题市场。

### 6.3.3 发布对象

公开发布对象为整个知识库，而不是单篇文档。

### 6.3.4 功能要求

- 用户可在知识库设置中开启或关闭公开发布
- 用户可配置公开知识库名称、简介、封面
- 系统自动生成公开访问地址
- 公开页必须包含左侧目录树
- 公开页必须支持目录导航、上一篇/下一篇切换
- 公开页必须支持站内文档搜索
- 公开页首页默认展示知识库简介与目录
- 公开页文档支持阅读模式，不显示后台管理操作

### 6.3.5 访问规则

- 未发布的知识库不可公开访问
- 已发布知识库可被匿名访问
- 若知识库关闭公开发布，公开地址立即失效
- 文档被删除或移入回收站后，不在公开站点中展示

### 6.3.6 与现有单篇分享的关系

- 单篇分享能力继续保留
- 知识库公开发布与单篇分享是两套独立入口
- 单篇分享适用于外链临时分享
- 知识库公开发布适用于长期公开文档站点

### 6.3.7 数据要求

需要补充或确认以下字段：

- 知识库表：`publish_enabled`
- 知识库表：`publish_slug`
- 知识库表：`publish_description`
- 知识库表：`publish_cover_url`

### 6.3.8 验收标准

- 用户开启发布后，可通过公开地址访问知识库首页
- 公开页可以按目录树浏览至少 50 篇文档
- 搜索结果只返回当前知识库下的公开文档
- 关闭发布后，原公开地址无法访问

## 6.4 FR-04 成员与权限

### 6.4.1 目标

让协作从“分享链接驱动”升级为“知识库成员驱动”。

### 6.4.2 权限层级

V1.0 采用知识库级权限模型，不做文档级细粒度授权。

支持角色如下：

- Owner
- Admin
- Editor
- Commenter
- Viewer

### 6.4.3 功能要求

- 知识库所有者可邀请成员加入知识库
- 所有者和管理员可修改成员角色
- 所有者可移除成员
- 成员列表页必须显示成员昵称、账号、角色、加入时间
- 被邀请成员接受邀请后，知识库出现在其首页列表中

### 6.4.4 权限规则

- Owner：拥有全部权限，包括删除知识库、管理成员、发布知识库
- Admin：可管理目录、文档、成员角色，但不可删除知识库，不可转移所有权
- Editor：可新建、编辑、移动、删除文档与目录，不可管理成员
- Commenter：可阅读并评论，不可编辑文档
- Viewer：仅可阅读

### 6.4.5 与现有系统角色的关系

- 现有用户系统角色 `USER / ADMIN` 保留，用于平台级管理
- 本需求新增的是“知识库成员角色”，两者不可混用
- 平台管理员不自动获得任意知识库编辑权限，除非加入该知识库或通过后台强制介入

### 6.4.6 业务约束

- 单知识库成员上限：100
- 同一用户在同一知识库下只能拥有一个角色
- 仅 Owner 可转移所有权
- Owner 不可将自己降级为非 Owner，除非先完成所有权转移

### 6.4.7 数据要求

需要新增知识库成员关系模型，建议字段如下：

- `knowledge_base_members.id`
- `knowledge_base_members.notebook_id`
- `knowledge_base_members.user_id`
- `knowledge_base_members.role`
- `knowledge_base_members.invite_status`
- `knowledge_base_members.joined_at`

### 6.4.8 接口要求

至少需要提供以下 API：

- 创建邀请
- 接受邀请
- 查询成员列表
- 修改成员角色
- 删除成员
- 查询当前用户在指定知识库下的成员角色

### 6.4.9 验收标准

- Owner 可成功邀请成员加入知识库
- 不同角色登录后，对同一知识库看到的操作权限不同
- Commenter 无法编辑文档但可发表评论
- Viewer 无法编辑也无法评论
- Admin 可管理成员，但不能删除知识库

## 7. 非功能需求

## 7.1 性能要求

- 打开包含 1000 个节点的目录树时，首屏可交互时间不超过 3 秒
- 公开知识库首页首次加载时间不超过 3 秒
- 批量导入过程中，前端必须有进度反馈，不允许长时间无状态

## 7.2 安全要求

- 公开知识库接口与私有知识库接口必须严格隔离
- 成员权限校验必须在服务端执行，前端仅作展示控制
- 导入文件必须校验类型和大小，禁止执行型文件上传

## 7.3 可用性要求

- 所有关键操作必须有明确成功/失败反馈
- 删除、关闭发布、移除成员等危险操作必须二次确认
- 导入失败结果必须可追踪，不允许只提示“导入失败”

## 8. 版本依赖关系

本版本建议按以下顺序实施：

1. FR-02 多级目录树  
2. FR-01 导入中心  
3. FR-04 成员与权限  
4. FR-03 知识库公开发布

实施原因如下：

- 多级目录树是导入结构落地的基础
- 成员与权限是公开发布前的数据访问基础
- 公开发布依赖目录树和权限边界完成后再落地更稳

## 9. 里程碑建议

### 里程碑 M1：目录树基础可用

- 完成文件夹与笔记的多级目录树
- 完成拖拽移动、展开折叠、面包屑
- 完成目录排序持久化

### 里程碑 M2：导入闭环可用

- 完成 Markdown、ZIP、Word 导入
- 完成导入资源处理
- 完成导入结果页与失败日志

### 里程碑 M3：成员协作可用

- 完成知识库成员模型
- 完成邀请、角色调整、成员页
- 完成服务端权限校验

### 里程碑 M4：知识库发布可用

- 完成公开知识库首页
- 完成公开目录树与阅读页
- 完成知识库级搜索与发布开关

## 10. 交付标准

当且仅当以下条件全部满足时，本版本视为完成：

- 四个功能模块均已完成开发、自测与联调
- 每个模块的验收标准均可通过手动验证
- 公开访问、成员权限、导入结果等关键路径均无阻断性问题
- 文档、接口、数据库变更已同步更新

## 11. 后续版本候选项

以下需求明确延后到后续版本评估：

- 双链、反向链接
- 评论通知中心
- 搜索权重优化与全文索引
- AI 深度编辑
- 自定义域名、主题样式
- 表格数据库和任务视图
